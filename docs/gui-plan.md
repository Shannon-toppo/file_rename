# file_rename GUI 版 実装計画

作成日: 2026-07-04 / 対象: `file_rename/`(mv2title コンシューマ)に GUI アプリを追加

## ゴール

- **Excel 風の操作感**を持つテーブル UI。1 行 = 1 曲(URL または既存ファイル)。
- 基本フローは全自動: **URL 入力 → yt-dlp で DL → mv2title でタイトル推定 → メタデータ書き込み**。
- ただし**手動修正**が可能: 推定タイトルのセル編集、選択行の再推定、書き込み前の確認モード。
- 既存 CLI(`rename.py` / `download.py`)は残す。共通ロジックは GUI と CLI で共用する。

## 技術選定(推奨案)

| 項目 | 推奨 | 理由 |
|---|---|---|
| GUI フレームワーク | **PySide6 (Qt)** | `QTableView` + Model/View が Excel 風グリッドの標準解。シグナル/スロットでワーカースレッド → UI 更新が安全に書ける。LGPL で配布も可 |
| グリッド | `QTableView` + カスタム `QAbstractTableModel` | 行の状態遷移(DL中/推定中/完了…)をモデル一元管理し、ソート・部分更新・編集可否をセル単位で制御できる |
| 並行処理 | `QThreadPool` + `QRunnable`(または `QThread` ワーカー) | yt-dlp / LLM 呼び出しはブロッキングなので UI スレッドから隔離。進捗はシグナルで通知 |
| テスト | `pytest-qt` | モデル・ワーカーのオフラインテスト(LLM はフェイク注入、yt-dlp はスタブ) |

**対抗案**: tkinter + `tksheet`(依存が軽く、コピペ・セル編集が組み込み済み。ただしスレッド連携が `after()` ポーリングになり、見た目も古い)。PySide6 の依存サイズ(数十 MB)が問題になる場合のみ採用。
→ **フェーズ 0 でユーザー確認して確定**(決定記録に残す)。

## 画面イメージ

```
[URL入力欄______________________] [追加] [ファイル追加]   [▶ 一括実行] [⏸] 
┌─┬────────────────────┬──────────┬──────────────┬────────┬──────┐
│#│ 元タイトル(ファイル名)   │ チャンネル │ 推定タイトル ✎  │ 状態    │ 形式  │
├─┼────────────────────┼──────────┼──────────────┼────────┼──────┤
│1│ Artist - Song [MV]...   │ ArtistCh  │ Song          │ ✅完了  │ mp3  │
│2│ ...                     │ ...       │ (編集可)       │ DL中 45%│ mp3  │
└─┴────────────────────┴──────────┴──────────────┴────────┴──────┘
[選択行を再推定] [選択行を書き込み] [行削除]     ☑ 自動書き込み   [設定]
```

- 行の状態遷移: `URL入力 → キュー → DL中 → 推定中 → (自動書き込み ON: 書き込み → 完了 / OFF: 確認待ち)`。失敗は `エラー`(ツールチップに詳細)。
- 「推定タイトル」列のみ編集可。手動編集した行は `手動` マークを付け、再推定で上書きしない(明示的な再推定は除く)。

## 委譲可否の凡例

Fable のトークン節約のため、各タスクにモデル委譲の可否を付記する(`../mv2title/docs/refactoring-plan.md` と同一の凡例)。

| 記号 | 意味 |
|---|---|
| 【S】 | **Sonnet に委譲可** — 機械的・仕様が本書で完結・テストが安全網になる |
| 【O】 | **Opus に委譲可** — 局所的な設計判断を含むが、本書の指示とテストで完結する |
| 【F】 | **Fable(メインセッション)で実施** — 横断的な設計判断・スレッド安全性・ユーザー判断が絡む |

**委譲の運用方針**:
- GUI はボイラープレートの塊(ウィジェット配置・ダイアログ・スタイル)であり、**委譲効率が非常に高い**。フェーズ 1 で Fable がインターフェース(モデルのカラム定義・ワーカーのシグネチャ・シグナル一覧)を固めれば、フェーズ 2 以降の実装の大半は【S】【O】で進められる。
- 委譲時は本書の該当フェーズ + 対象ファイルだけをコンテキストに渡せば足りるよう、各フェーズに「成果物」を明記した。
- 【O】タスクはマージ前に Fable が差分レビュー(実装より格段に安い)。スレッド境界(シグナル発行・モデル更新)に触る変更だけは【F】でレビュー必須。

## フェーズ 1: コアの GUI 非依存化(最重要) ✅ 実施済み (2026-07-04)

現状のコアは `print()` に結果を流す設計で GUI から使えない。**進捗・結果をコールバック/戻り値で返す層**に作り替える。CLI もこの層の上に乗せ直す。ここの API 設計が全フェーズの土台。

| タスク | 委譲 | 状態 |
|---|---|---|
| コア API の設計: `core.py` に `Track` dataclass + `Status` enum と、`download_tracks(url, fmt, on_progress, cancel)` / `infer_titles(tracks, client, force)` / `write_tags(tracks, on_result)` を定義。キャンセルは `threading.Event` | 【F】 | ✅ スキップ方針(空タイトル/検証失敗は書かない、手動編集済みは書く)も write_tags に集約 |
| `download.py` の DL ロジックを `core` へ移設し、`print` を進捗コールバックへ置換(yt-dlp の `progress_hooks` で DL% も取得)。CLI(`download.py` / `rename.py`)を core の薄いラッパに書き直し | 【O】 | ✅ Fable が直接実施(core 設計と一体のため)。CLI 表示は `core.describe_result` で共通化 |

| core のオフラインテスト(フェイク LLMClient・yt-dlp スタブ・一時ファイルへのタグ書き込み) | 【S】 | ✅ `tests/test_core.py` 14 件全緑(手動保護・件数不一致・キャンセル・スキップ方針を網羅) |

## リスクと順序の根拠

- **最大のリスクはスレッド境界**(ワーカー → モデル更新)。ここを間違えると再現困難なクラッシュになるため、フェーズ 0 のスパイクで疎通パターンを確立し、フェーズ 2 のワーカー設計は【F】で行う。以降のフェーズはそのパターンを踏襲するだけなので委譲できる。
- **フェーズ 1 を GUI より先に行う理由**: `print` ベースのままウィジェットを作り始めると、後で全ワーカーを書き直すことになる。core の API が固まれば、GUI 側は「シグナルで受けて表に映す」だけの委譲しやすい仕事になる。
- **バッチ推定との整合**: mv2title はバッチ前提(1 リクエスト N 件)なので、行ごとに LLM を叩く設計にしない。キューに溜まった行をまとめて推定する(フェーズ 2)。
- mv2title 本体には**変更不要**の見込み(公開 API `extract_titles` / `TitleInput` で足りる)。もし不足(例: 進捗コールバック)が出たら mv2title 側に issue として起こし、この計画とは分離する。
